# ADR-010: Ingest Agent — Standalone Process and LLM-Driven Processing Pipeline

**Status:** Accepted

**Date:** 2026-06-24

## Context and Problem Statement

The Ingest Agent (feature 004) is the first agent to exercise the container escape-hatch defined in ADR-002. Two distinct decisions must be made explicit:

1. **Deployment topology**: The agent spec requires standalone CLI operation, independent containerizability, and a deployment lifecycle separate from the Hub. This disqualifies the default in-process Worker Service path.
2. **Processing pipeline**: The pipeline must analyse, structure, and index raw content. No vector embedding infrastructure exists or is planned for the project. The Claude Code SDK (Anthropic) is available and provides richer content understanding than a classical embedding service.

These two concerns are recorded in a single ADR because the LLM dependency is a direct consequence of placing the agent outside the .NET backend process — it brings its own runtime and dependency chain.

## Decision Drivers

- ADR-002 escape-hatch: agent requires standalone CLI mode and independent containerization
- ADR-005 monorepo layout already reserves `src/agents/` for this case
- No vector embedding service is available or planned in current infrastructure
- Claude SDK provides content analysis, chunking guidance, metadata extraction, and structured output without additional infrastructure
- Standalone process enables development and testing without the full platform
- IAgentWorker contract (ADR-002) is fulfilled via the Hub's RemoteAgent HTTP proxy, not implemented inside the agent itself

## Considered Options

### Option A: In-process Worker Service in `src/backend/` (ADR-002 standard path)

**Rejected.** Conflicts with the standalone and containerizable deployment requirement. Embedding the agent inside the backend process makes independent CLI operation impossible and couples the agent's lifecycle to the Hub's.

### Option B: Standalone process in `src/agents/ingest/` with Claude SDK ← chosen

The agent is a self-contained runnable process. It communicates with the Hub over HTTP when available and falls back to CLI-only mode otherwise. Content processing uses the Claude SDK as the sole AI processing dependency.

### Option C: Python microservice with a vector embedding model

**Rejected.** Introduces Python runtime and embedding model infrastructure with no justification — the project has no retrieval pipeline that requires vector similarity search at this stage. Adds operational complexity without benefit.

## Decision Outcome

**Chosen: Option B — Standalone process in `src/agents/ingest/`, LLM-driven pipeline via Claude SDK.**

### Source Location

```
src/agents/ingest/          # Standalone agent process
```

The agent is a first-class project member under `src/agents/`, consistent with ADR-005's reserved directory and peer to `src/backend/` and `src/frontend/`.

### Hub Integration

The Hub registers the Ingest Agent via the `RemoteAgent` proxy path defined in ADR-002. The Hub holds an HTTP client pointing to the agent's endpoint; the `IAgentWorker` interface is never imported into the agent process itself.

```
Hub (src/backend/)
  └── RemoteAgent (HTTP proxy)
        └── Ingest Agent (src/agents/ingest/)   ← this ADR
```

### Processing Pipeline

Raw source files are processed through a sequential, LLM-driven pipeline:

```
Raw Source
  → Chunking (content-type-aware splitting)
  → Claude SDK (analysis: structure extraction, metadata, key concepts)
  → Structured Output (written to wiki/ or index store per ADR-007)
  → Git Commit (per ADR-007)
```

The Claude SDK replaces what a classical embedding-then-index pipeline would do. There is no separate embedding service.

### Configuration

All sensitive and environment-specific values are injected via environment variables:

| Variable | Purpose |
|----------|---------|
| `ANTHROPIC_API_KEY` | Claude API authentication (MUST NOT be hardcoded) |
| `ANTHROPIC_MODEL` | Model identifier (defaults to latest stable Sonnet) |
| `INGEST_SOURCE_DIR` | Path to `raw/sources/` (default: `./raw/sources`) |
| `INGEST_GIT_CREDENTIALS` | Git credentials for auto-commit |
| `INGEST_HUB_URL` | Hub HTTP endpoint (optional — omit for standalone mode) |
| `INGEST_FILE_SIZE_LIMIT_MB` | Feedback Request threshold in MB (default: 10) |

## Consequences

### Positive

- No embedding or vector database infrastructure required
- Content analysis quality from Claude exceeds keyword/embedding approaches for structured knowledge extraction
- Independent process: fully testable via CLI without Hub or frontend
- Aligns with ADR-002 escape-hatch design — no architectural rework needed
- ADR-005 `src/agents/` directory was reserved precisely for this

### Negative

- External API dependency on Anthropic Claude API — requires outbound internet connectivity during processing
- Per-run API cost proportional to source volume
- Rate limiting from the Anthropic API must be handled (retry with backoff)
- LLM responses require output validation — malformed structured output must be detected and logged as `failed`, not silently accepted

## Architecture Test

Per the constitution (Principle III), an automated architecture test MUST enforce:

- The `src/agents/ingest/` project MUST NOT import any namespace from `src/backend/`
- `ANTHROPIC_API_KEY` MUST NOT appear as a literal string in any source file (secret scanning gate)

## Notes

**Offline / standalone mode:** When `INGEST_HUB_URL` is not set, the agent runs without Hub registration. When `ANTHROPIC_API_KEY` is absent or the API is unreachable, LLM calls fail gracefully — the affected file's `IngestRecord` is set to `failed` with the error recorded; the run continues for remaining files.

**Model pinning:** `ANTHROPIC_MODEL` defaults to the latest stable Claude Sonnet. The specific model is a deployment concern, not an architectural one, and MUST NOT be hardcoded.

**Relationship to ADRs:** This decision extends ADR-002 (escape-hatch activation), ADR-005 (monorepo `src/agents/` slot), ADR-006 (RemoteAgent proxy path), and ADR-007 (Git commit output).
