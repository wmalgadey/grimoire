# ADR-002: Agent Runtime Strategy — Worker Services Instead of Container-per-Agent

**Status:** Accepted

**Date:** 2026-06

## Context and Problem Statement

Agents should support various use cases and could require different technologies. Initial thinking was container-per-agent (maximum isolation, arbitrary languages). The project is solo-developed, Spec-Kit-first.

## Decision Drivers

- Operational overhead must match project scope
- Migration path to container isolation must remain open
- Contract-first approach fits better with SDD: specs define the interface, not the implementation
- Local development and debugging must be practical

## Considered Options

1. **Container-per-Agent (Docker/Podman)**
2. **.NET Worker Services + Interface Abstraction** ← chosen
3. gRPC-based sidecar processes

## Decision Outcome

**Chosen: .NET Worker Services with IAgentWorker Interface + Future Container Escape-Hatch**

The interface is defined in the spec, Worker Services implement it inline initially. If an agent requires another language or isolation, the interface is exposed via HTTP/gRPC externally — no architectural rework. Container-per-agent becomes a deployment detail, not an architectural decision.

```
IAgentWorker
├── InProcessAgent (Worker Service, .NET)
└── RemoteAgent (HTTP/gRPC Proxy → arbitrary container)
```

## Consequences

### Positive

- No orchestration complexity in early phases
- Simple local debugging
- SDD specs can focus on the interface, not infrastructure
- Migration path to true container isolation remains open

### Negative

- Initially no true language-mixing for agents
- Worker Service process crash affects all agents (until containerization)

## Notes

**Trigger for container migration:** If an agent requires Python ML libraries (e.g., direct NumPy/Transformers integration) or fault isolation becomes critical.

**Amendment:** The `RemoteAgent` proxy concept described above was revisited when the first standalone agent was introduced. See [ADR-010](adr-010.md) for the refined communication pattern between Hub and standalone agents.
