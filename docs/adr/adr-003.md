# ADR-003: Frontend — Svelte 5 + Vite Instead of HTMX

**Status:** Accepted

**Date:** 2026-06

## Context and Problem Statement

The Web UI is the central channel with use-case-specific components. Initial thinking was HTMX as a lightweight alternative to React/Svelte. The UI must handle streaming agent responses, in-flight agent tasks, and real-time updates.

## Decision Drivers

- Streaming agent responses (SSE/WebSocket) must be rendered client-side
- Component-based reuse for use-case-specific UI blocks
- SDD agents generate React/Svelte components better than HTMX partials
- Bundle size and learning curve should remain low

## Considered Options

1. **HTMX**
2. **Svelte 5 + Vite** ← chosen
3. React 19 + Vite

## Decision Outcome

**Chosen: Svelte 5 + Vite**

HTMX is optimized for server-rendered-HTML flows — a model that structurally conflicts rather than harmonizes with streaming agent responses and complex client-side state (in-flight tasks, session context). Svelte provides reactive components without Virtual DOM overhead, small bundle, and is highly specifiable for SDD agents. React would be valid, but Svelte is leaner and has less boilerplate, benefiting code generation quality.

## Consequences

### Positive

- Reactive components for streaming/real-time natively
- Small bundle, no Virtual DOM
- Compact component code — less noise in agent-generated output
- Vite as build tool is de-facto standard, broad agent support

### Negative

- Svelte 5 Runes syntax is newer paradigm — LLM training data may be heavier on Svelte 4
- Smaller ecosystem than React (UI libraries, community components)

## Notes

If Svelte-5-specific code generation issues arise: explicitly reference Svelte 5 Runes syntax in `/speckit.constitution` and include example components in context.
