# ADR-007: Storage Strategy — Git + Plain Markdown Files

**Status:** Accepted

**Date:** 2026-06

## Context and Problem Statement

Grimoire stores user knowledge (wiki pages, audit findings, batch syntheses) as persistent state. The storage layer must be:
- Git-native (version control built-in, diffable, mergeable)
- Obsidian-compatible (users can browse/edit in Obsidian)
- Human-readable (no opaque formats, fully debuggable)
- Single-file-per-entity (enables atomic commits, clear ownership)

Initial inspiration: Andrej Karpathy's ["LLM.txt"](https://github.com/karpathy/LLM.md) — the idea that a single, human-curated Markdown file with well-structured content can serve as a knowledge base.

## Decision Drivers

- Knowledge must be portable and readable outside Grimoire
- Git provides natural audit trail and rollback capability
- No binary dependencies (database, vector store) in the happy path
- Developers and users should be able to inspect/edit raw files
- Semantic Search is optional (can add vector index later without schema change)

## Considered Options

1. **Plain Markdown in Git** ← chosen
2. JSON/YAML in Git (structured, less human-readable)
3. Traditional database (PostgreSQL, SQLite) with Git export
4. Hybrid: Git for authoring, vector database for query (Qdrant, Weaviate)

## Decision Outcome

**Chosen: Plain Markdown Files + Git as Primary State Layer**

All persistent state is stored as plain Markdown files committed to a Git repository:

```
repo/
├── wiki/              # User-facing knowledge pages
│   ├── topic-a.md
│   ├── topic-b.md
│   └── batch/         # Agent-generated syntheses
│       └── connections-2026-06-23.md
├── audits/
│   ├── open/          # Unresolved findings
│   │   └── orphan-pages-20260623.md
│   └── resolved/      # Audit trail
│       └── orphan-pages-20260623-resolved.md
└── raw/sources/       # Agent input folder
    └── (watched by Ingest)
```

Each file is a self-contained Markdown document with:
- YAML front-matter for metadata (creation date, author, tags)
- Human-readable body
- Optional structured fields (e.g., JSON blocks for audit findings)

Commits capture every mutation:
```
ingest: sync 2026-06-23
lint: 3 orphans, 2 contradictions
batch: weekly syntheses
resolve: orphan "foo" → page created
```

## Consequences

### Positive

- **Portability**: Users can clone the repo and use it in any Markdown editor (Obsidian, Logseq, Vim)
- **Auditability**: `git log` is the audit trail; `git blame` tracks authorship
- **Debuggability**: Any issue can be inspected by reading raw files
- **Diffability**: Pull requests on Markdown are human-readable, enabling peer review
- **Single source of truth**: No schema sync issues between services
- **Simplicity**: No database server, migrations, or ORM to maintain
- **Integration**: Agents read/write files directly; no ORM layer needed

### Negative

- **Query performance**: Semantic search requires indexing (e.g., offline vector index or eventual consistency with external store)
- **Scalability**: Very large wikis (millions of pages) may have slow Git operations
- **Transactions**: No ACID guarantees; concurrent writes can conflict (mitigated by work-queue patterns in agents)
- **Complexity of aggregations**: Computing global statistics requires scanning files

## Mitigation Strategies

**Semantic Search (optional future):**
- Batch indexing job reads all files → vector database (Qdrant, Weaviate)
- Index is read-only, not authoritative — regenerated on demand
- Markdown files remain the source of truth

**Concurrency:**
- Agents serialize work via a request queue (Hub orchestrator, see ADR-006)
- No two agents write to the same file simultaneously
- Lint and Batch are read-only (safe to run in parallel)

**Scale:**
- For very large wikis (millions of pages), split into multiple Git repos by domain (see Bounded Contexts in Ubiquitous Language)
- Each agent owns its subrepository

## Notes

**Related ADRs:** ADR-005 (monorepo structure) specifies how this storage integrates with the project layout. Agents (ADR-002) interact with storage via file system operations; the IAgentWorker interface is storage-agnostic, allowing future migration to database if needed.

**Inspiration:** Karpathy's LLM.txt demonstrates that minimalist, human-readable documentation can be more valuable than elaborate structured schemas. This ADR applies that principle to the entire knowledge base.
