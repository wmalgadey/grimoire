# ADR-008: Operational State Persistence — SQLite for Hub Ephemera

**Status:** Accepted

**Date:** 2026-06

## Context and Problem Statement

The Hub must persist two categories of state:

1. **Domain State**: User knowledge (wiki pages, audit findings, batch syntheses) — permanent, user-facing, version-controlled
2. **Operational State**: Agent job queues, agent status, execution history — ephemeral, internal bookkeeping, tied to a specific Hub instance

ADR-007 establishes Git + Markdown for domain state. This ADR addresses operational state, which has different durability, consistency, and lifecycle requirements.

## Decision Drivers

- Domain state must survive Hub shutdown/restart and be user-auditable (Git + Markdown, per ADR-007)
- Operational state must survive unplanned Hub restarts (logs of which jobs ran, job status) but is not user-facing
- SQLite is embedded (no external database server) — matches early project stage and solo development
- Operational state and domain state must not pollute each other (no SQLite writes to wiki/, no Git commits for job metadata)

## Considered Options

1. **SQLite for operational state only, Git+Markdown for domain state** ← chosen
2. SQLite for all state (operational + domain) — violates ADR-007 principle of Git-native content
3. In-memory with periodic file dumps — loses state on unplanned shutdown
4. Separate PostgreSQL database — too much infrastructure for single-developer project at this stage

## Decision Outcome

**Chosen: SQLite for Operational State, Git+Markdown for Domain State**

The Hub maintains two independent persistence layers:

```
Hub (Minimal API)
├── SQLite database (operational state)
│   ├── AgentDescriptors table
│   ├── AgentJobs table
│   └── AgentHealthEvents table
└── Git repository (domain state)
    ├── wiki/ (user knowledge)
    ├── audits/ (audit findings)
    └── raw/sources/ (ingest input)
```

**Operational State (SQLite):**
- Agent descriptors and lifecycle status
- Job queue and execution history
- Health check events
- Database lifecycle: created on startup if missing, auto-schema generated
- File path: configurable via appsettings.json or environment variable

**Domain State (Git+Markdown):**
- All user-created and agent-generated knowledge
- Agent write operations update Git (via git add/commit) — never via SQLite
- Lint and Batch agents commit audit findings directly to Git

## Consequences

### Positive

- **Separation of concerns**: Operational metadata doesn't clutter the user-facing knowledge repo
- **No schema migrations needed**: SQLite schema is auto-created per ADR requirements
- **Resilience**: Job queue survives unplanned shutdown; domain state is auditable via Git
- **Debuggability**: `sqlite3 grimoire.db` can inspect operational state independently
- **Future scalability**: SQLite remains suitable for multi-Hub scenarios (each Hub has its own database)

### Negative

- **Two systems to manage**: Developers must reason about two persistence layers
- **Limited queries**: SQLite constraints vs full-featured databases mean complex aggregations are harder (mitigated by importing to analytics tools if needed)
- **No distributed transactions**: Hub-level failures can leave SQLite in intermediate state (mitigated by idempotent job processing)

## Mitigation Strategies

**Data Loss:**
- All SQLite writes are idempotent (safe to replay)
- Job status is the source of truth; if Hub crashes mid-job, job is retried on recovery

**Pollution:**
- Architecture tests enforce that no `wiki/`, `audits/`, or `raw/` directories are touched by SQLite operations
- Lint and Batch agents must commit to Git directly, never via SQLite

**Schema Evolution:**
- SQLite schema is versioned; migrations are applied on startup before feature code runs
- Schema versioning table tracks applied migrations

## Notes

**Related ADRs:**
- ADR-007 (Git + Markdown for domain state) — establishes that user-facing knowledge is never in databases
- ADR-006 (Hub-Spoke) — Hub orchestrates all state transitions; agents are passive
- ADR-002 (Worker Services) — agent registration and lifecycle status lives in Hub's SQLite

**Future**: If Hub instances need to share operational state (cluster scenario), SQLite is replaced with a shared store (PostgreSQL, etcd) without changing the Hub-Spoke architecture — the interface remains the same.
