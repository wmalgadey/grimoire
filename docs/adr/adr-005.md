# ADR-005: Monorepo Structure and Spec-Kit Workflow Integration

**Status:** Accepted

**Date:** 2026-06

## Context and Problem Statement

Backend (.NET), frontend (Svelte), and future agent implementations should live in a single monorepo but be independently developable. Spec-Kit is deployed as the SDD toolchain — the directory structure must support the Spec-Kit workflow.

## Decision Drivers

- No code mixing between layers
- Spec-Kit artifacts (`specs/`, `plans/`, `tasks/`) must clearly map to respective subprojects
- CI/CD pipelines should only build affected subprojects
- `constitution.md` must enforce project-wide constraints (stack, conventions, security)

## Considered Options

1. **Flat Monorepo with Directory Layout** ← chosen
2. Nx-based monorepo
3. Separate repositories

## Decision Outcome

**Chosen: Flat Monorepo with Explicit Directory Layout and Per-Subproject Spec-Kit Integration**

Nx is justified with team + CI caching requirements. For solo development with SDD, the overhead is unjustified. Separate repos fragment Spec-Kit context — the agent should draw project-wide context from a single `constitution.md`.

```
/
├── .specify/               # Spec-Kit root (constitution, templates, agent commands)
├── constitution.md         # Project-wide non-negotiables (stack, security, conventions)
├── specs/                  # Spec-Kit artifacts (per feature/component)
│   ├── backend/
│   ├── frontend/
│   └── channels/
├── src/
│   ├── backend/            # .NET 9 solution
│   ├── frontend/           # Svelte 5 app
│   └── agents/             # Agent worker implementations
├── docs/
│   └── adr/                # Architecture decision records
└── .github/
    └── workflows/          # Separate pipelines per src/ subdir
```

### constitution.md Required Contents (Spec-Kit)

- Tech stack constraints (reference these ADRs)
- C# coding conventions (nullable enabled, records for DTOs, async-first)
- Svelte 5 Runes syntax as standard
- `IChannel`/`IAgentWorker` as core interfaces — never bypass
- Security: no hardcoded secrets, all inputs validated

## Consequences

### Positive

- Spec-Kit agent has project-wide context via `constitution.md`
- Clear boundaries without tooling overhead
- Simple CI filtering via path-based triggers

### Negative

- No automatic dependency tracking between subprojects (manual or scripted)
- Retrofit Nx if team size or CI complexity demands it
