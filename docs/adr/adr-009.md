# ADR-009: Domain-Driven Code Organization — Screaming Architecture

**Status:** Proposed (for acceptance before proceeding with implementation)

**Date:** 2026-06-23

## Context and Problem Statement

The current `.NET Grimoire.Api` codebase is organized by technical layers: `/Api`, `/Core`, `/Infrastructure`. This layered architecture obscures the business domain. When a developer opens the project, they see technical concerns (endpoints, handlers, services) instead of business domains (Agents, Hubs, Channels).

This violates the project constitution principle **I. Domain Architecture & Strategic DDD**: the codebase does not immediately reveal its business purpose.

## Decision Drivers

- **Principle I Compliance**: Constitution mandates Domain Architecture & Strategic DDD be "applied from the first commit." Current structure violates this.
- **Developer Productivity**: New developers cannot quickly locate domain-specific code; architectural navigation requires jumping between multiple layer folders.
- **Code Locality**: Adding a feature for a specific domain requires modifying files across `/Api`, `/Core`, `/Infrastructure`, violating co-location and increasing context switches.
- **Maintainability**: As the system grows (more agents, channels), layer-based organization becomes increasingly fragmented.
- **Architectural Clarity**: The structure should "scream" the business intent (Hub-Spoke orchestration pattern from ADR-006, with Agents, Hubs, Channels as primary domains).

## Considered Options

1. **Domain-Driven Code Organization (Screaming Architecture)** ← chosen
   - Organize by business domain: `/Agents`, `/Hubs`, `/Channels`, `/Shared`
   - Each domain folder contains its own endpoints, handlers, domain services, and tests
   - Shared infrastructure (middleware, observability, persistence, exceptions) centralized in `/Shared`
   - Namespaces reflect domain organization: `Grimoire.Api.Agents.*`, `Grimoire.Api.Hubs.*`, etc.

2. Keep current layer-based structure (rejected)
   - Violates constitution principle I
   - Remains unclear for new developers
   - Does not support future scaling (each new domain compounds the navigation complexity)

3. Hybrid approach (rejected)
   - Layers for some code, domains for others
   - Creates inconsistency and increases confusion
   - Requires architectural governance overhead to maintain boundaries

## Decision Outcome

**Chosen: Domain-Driven Code Organization (Screaming Architecture)**

Reorganize `/src/backend/Grimoire.Api` from layer-based to domain-based structure:

```
src/backend/Grimoire.Api/
├── Agents/
│   ├── Endpoints/
│   ├── Handlers/
│   ├── Services/
│   ├── Models/
│   └── Tests/
│
├── Hubs/
│   ├── Endpoints/           (SignalR hub definitions)
│   ├── Handlers/            (Hub event handlers)
│   ├── Services/
│   ├── Models/
│   └── Tests/
│
├── Channels/
│   ├── Endpoints/
│   ├── Handlers/
│   ├── Services/
│   ├── Models/
│   └── Tests/
│
└── Shared/
    ├── Middleware/
    ├── Observability/
    ├── Persistence/
    ├── Exceptions/
    └── Models/
```

**Key Constraints:**

- Each domain folder is self-contained: developers work within a single folder for complete domain features
- **No circular dependencies between domains** (enforced by automated architecture tests)
- **Cross-domain communication via well-defined interfaces** (e.g., IAgentWorker from ADR-002, IChannel from ADR-004)
- Shared infrastructure available to all domains but centralized to prevent duplication
- Namespaces strictly follow domain organization: `Grimoire.Api.Agents.*`, `Grimoire.Api.Hubs.*`, `Grimoire.Api.Channels.*`, `Grimoire.Api.Shared.*`

## Consequences

### Positive

- **Principle I Compliance**: Code organization directly reflects business domains (Agents, Hubs, Channels)
- **Developer Productivity**: New developers can locate domain-specific code in 30 seconds; feature work stays within a single folder
- **Code Locality**: Features are implemented with minimal context switching; co-located code improves readability
- **Scalability**: New domains can be added by creating a new top-level folder without structural reorganization
- **Adherence to ADR-006**: The Hub-Spoke orchestration pattern becomes visible in the structure (Hub ↔ Agents, Hubs, Channels)
- **Testability**: Domain-specific tests are naturally isolated; integration tests verify cross-domain interactions

### Negative

- **One-time Refactor Cost**: Significant file movement and namespace updates required (mitigated by automated tooling)
- **Namespace Changes**: All imports must be updated to reflect new namespace organization
- **Test Structure Reorganization**: Test project must be restructured to mirror domain organization

## Validation

- **Automated Architecture Test** (NetArchTest.Rules or equivalent): Enforce no circular dependencies between domains
- **Namespace Compliance Test**: Verify all types follow `Grimoire.Api.{Domain}.*` or `Grimoire.Api.Shared.*` pattern
- **CI/CD Gate**: All tests must pass; namespace and architecture rules must be validated before merge

## Relationship to Other ADRs

- **ADR-005** (Monorepo Structure): This decision improves the domain-level organization within the backend subproject
- **ADR-006** (Hub-Spoke Orchestration): This structure makes the Hub-Spoke pattern visible and enforceable at the filesystem level
- **Constitution Principle I**: This decision fulfills "Domain Architecture & Strategic DDD" requirement by organizing code by business domain

## Notes

This refactoring is **purely architectural**: no new features are added, no breaking changes to APIs, and all test logic remains unchanged. The refactor succeeds when all existing tests pass with zero modifications to test logic itself.
