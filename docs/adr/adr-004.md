# ADR-004: Channel Abstraction — Unified Channel Interface in Backend

**Status:** Accepted

**Date:** 2026-06

## Context and Problem Statement

Web UI and Telegram are initial channels. Additional channels should be pluggable without architectural rework. Channels differ in protocol (HTTP/SSE, Telegram Bot API), display capabilities, and interaction model.

## Decision Drivers

- New channels must be addable without core changes (Open/Closed Principle)
- Business logic must not leak into channel code
- SDD specs should clearly define channel contracts

## Considered Options

1. **Ad-hoc channel implementations without shared interface**
2. **IChannel Abstraction with Channel Registry** ← chosen
3. Message-broker-based (NATS, RabbitMQ)

## Decision Outcome

**Chosen: IChannel Abstraction with Channel Registry in Backend**

An explicit interface (`IChannel`) separates protocol details from orchestration logic. The registry enables dynamic routing of agent outputs to active channels. Message broker is overkill for current project scope — can be retrofitted later if horizontal scaling becomes relevant.

```
IChannel
├── WebUiChannel      (SignalR Hub)
├── TelegramChannel   (Telegram Bot API)
└── [Future channels]

OrchestratorService → ChannelRegistry → IChannel[]
```

## Consequences

### Positive

- Clear spec boundaries: each channel has its own spec file in Spec-Kit
- Testability: channels are independently mockable
- Addability without core changes

### Negative

- Initial interface design must be carefully captured in the spec (poorly specified interface → affects all channel implementations)
