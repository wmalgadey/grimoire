# ADR-006: Hub-Spoke Orchestration Architecture

**Status:** Accepted

**Date:** 2026-06

**Amended by:** [ADR-010](adr-010.md) — standalone agents extend the agent model

## Context and Problem Statement

Grimoire requires a system architecture that routes user requests through a central orchestrator to multiple specialized agents (Ingest, Query, Lint, Batch). Each agent has distinct responsibilities but all must coordinate through a single backend entry point.

## Decision Drivers

- Multiple agents with distinct domains must coexist without coupling
- User requests (Web UI, Telegram) arrive at a single backend
- Agents must be independently testable and replaceable (see ADR-002)
- Clear isolation of concerns: orchestration logic separate from agent logic

## Considered Options

1. **Hub-Spoke Orchestration** ← chosen
2. Message-broker-centric (agents subscribe to topics)
3. Direct agent-to-agent communication (choreography)

## Decision Outcome

**Chosen: Hub-Spoke Orchestration Pattern**

The backend (Hub) acts as a central orchestrator that:
- Routes incoming requests from channels (Web UI, Telegram, future channels)
- Dispatches work to appropriate agents (Spokes) via `IAgentWorker` interface (ADR-002)
- Collects results from agents and broadcasts via `IChannel` (ADR-004)
- Maintains request/response state during agent execution

**In-process agents are passive**: they implement a single interface and react to work assignments dispatched by the Hub. No in-process agent initiates work autonomously or communicates with other agents directly.

**Standalone agents extend this model** (see ADR-010): a standalone agent process runs outside the Hub, may initiate work autonomously (e.g., via file watcher), and communicates with the Hub over HTTP. The Hub remains the central relay between channels and agents — it dispatches commands to the agent and receives event callbacks from it. The agent does not communicate with channels directly.

```
User Inputs
  ↓
Channels (Web UI, Telegram, etc.) ← IChannel (ADR-004)
  ↓
Hub/Orchestrator (Backend)
  ↓
Agent Workers
  ├── In-process: IAgentWorker (ADR-002) — passive, Hub-driven
  │   ├── Query (wiki/ → response)
  │   ├── Lint  (wiki/ → audits/)
  │   └── Batch (wiki/ → syntheses)
  └── Standalone: HTTP API (ADR-010) — autonomous, Hub-relayed
      └── Ingest (raw/sources/ → wiki/)
  ↓
Results → Back to User via IChannel
```

## Consequences

### Positive

- Clear separation of concerns: orchestration logic decoupled from agent logic
- Each agent can be independently tested, replaced, or scaled (see ADR-002)
- New agents can be added without modifying the Hub (Open/Closed Principle)
- Request tracing and observability naturally flow through the Hub
- Works with both in-process (Worker Services) and standalone (HTTP) agents

### Negative

- Hub becomes a potential bottleneck if agent load is very high
- Complex request orchestrations (multi-agent workflows) must be explicit in the Hub
- Standalone agents introduce async event flows (callback-based) alongside synchronous Hub-dispatched commands — developers must reason about both patterns

## Notes

**Future scaling:** If the Hub becomes a bottleneck, the `IChannel` and `IAgentWorker` interfaces remain abstractions — async messaging (NATS, RabbitMQ) can replace direct calls without architectural rework. The Hub-Spoke pattern is independent of inter-process communication mechanism.

**Relationship to ADRs:** This pattern is enabled by ADR-002 (Worker Services with `IAgentWorker`), ADR-004 (`IChannel` abstraction), and realized in ADR-005 (monorepo structure where the backend orchestrator is the primary .NET service). ADR-010 extends this pattern for standalone agents.
